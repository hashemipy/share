# راهنمای Sync State Tracking

## مشکل قدیم (حل شده)
گاهی موجودی تغیر شده اعمال نمی‌شد و به حالت قبلی برمی‌گشت.

**دلیل:** استفاده از Timestamp برای مقایسه - اما Timezone و Server Time Sync معتبر نیستند.

## راه‌حل جدید: Sync State Tracking

### ستون‌های جدید در جدول `inventory_sync_mapping`:
```sql
last_synced_stock_site1 INT DEFAULT 0
last_synced_stock_site2 INT DEFAULT 0
last_synced_timestamp DATETIME
```

### منطق کار:

هر sync operation این را بررسی می‌کند:

```
current_stock_site1 = موجودی فعلی سایت 1
last_synced_stock_site1 = آخرین موجودی که sync شد (سایت 1)

اگر current_stock_site1 ≠ last_synced_stock_site1
    → سایت 1 تغیر کرده است
```

### سناریوها:

#### 1️⃣ فقط سایت 1 تغیر کرده
- سایت 1: 100 → 80
- سایت 2: 100 (بدون تغیر)
- **نتیجه:** موجودی سایت 2 به 80 برود

#### 2️⃣ فقط سایت 2 تغیر کرده
- سایت 1: 100 (بدون تغیر)
- سایت 2: 100 → 70
- **نتیجه:** موجودی سایت 1 به 70 برود

#### 3️⃣ هر دو سایت تغیر کردند (Conflict)
- سایت 1: 100 → 80
- سایت 2: 100 → 90
- **نتیجه:** بیشترین انتخاب شود = 90

### مثال عملی:

**مرحله 1: اولین Sync**
```
سایت 1: 100 | سایت 2: 100
last_synced_stock_site1 = 100
last_synced_stock_site2 = 100
```

**مرحله 2: کاربر موجودی سایت 1 را به 80 تغیر می‌دهد**
```
سایت 1: 80 | سایت 2: 100

Check:
- 80 ≠ 100 (last_synced_site1) → سایت 1 تغیر کرده
- 100 = 100 (last_synced_site2) → سایت 2 نتغیر کرده

نتیجه: سایت 2 را به 80 برود
```

**مرحله 3: بعد از Sync**
```
سایت 1: 80 | سایت 2: 80
last_synced_stock_site1 = 80
last_synced_stock_site2 = 80
```

## تفاوت با راه‌حل قدیم

| خصوصیت | قدیم | جدید |
|--------|------|------|
| منبع تصمیم | Timestamp | Sync State |
| قابلیت اعتماد | ضعیف | ۱۰۰٪ |
| Timezone مشکل | بله | خیر |
| وقتی هر دو تغیر کنند | نامعلوم | واضح (بیشترین) |
